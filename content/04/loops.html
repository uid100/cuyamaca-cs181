C++ Loops

Using for, while, and do-while in C++.

Watch for: Start • Stop • Step

Overview

Loops repeat a block of code. Every loop can be described with three ideas:

  • Start: the initial state (e.g., a counter's first value)
  • Stop: the condition checked each iteration (when this becomes false, the loop ends)
  • Step: how we move toward stopping (e.g., increment/decrement)

Pick the loop that best matches how clearly you can express Start, Stop, and Step.

The for Loop

Use a for loop when you know (or can easily express) the Start, Stop, and Step right up front.

Or, put another way, when you know how many times you want to repeat the loop.

Shape

//       Start    Stop     Step
for (int i = 0;   i < 5;      i++) {
    std::cout << i << " ";
}
// Output: 0 1 2 3 4

 

  • Start: int i = 0; — initialize counter
  • Stop: i < 5; — loop runs while true
  • Step: i++ — move toward stopping

Common Variations

// Count down
for (int i = 5; i > 0; --i) {
    // Start: 5, Stop: i > 0, Step: --i
}

// Custom step
for (int i = 0; i <= 10; i += 2) {
    // Start: 0, Stop: i <= 10, Step: +2
}

Watch for:

  • Forgetting to update the counter (infinite loop risk)
  • Using the wrong comparison (off-by-one errors)
  • Mutating the loop variable in the body unexpectedly

When is a for loop more readable than a while loop?

The while Loop

Use while when you want to keep looping as long as some condition holds true, and the Step might depend on work done in the loop body.

Shape

// Start (set up before the loop)
int count = 0;

// Stop (checked before each iteration)
// Step (performed in the body)
while (count < 5) {
    std::cout << count << " ";
    count++; // Step
}
// Output: 0 1 2 3 4

Annotated

  • Start: initialize variables before the loop
  • Stop: while (condition) — loop runs while true
  • Step: must happen in the body (e.g., count++)

Use Cases

  • Reading input until EOF or a sentinel value
  • Waiting for a condition to change
  • When iteration count isn't known upfront
Watch for:
  • Forgetting the Step → infinite loop
  • Condition never becomes false
  • Modifying the condition variable incorrectly

int total = 0;
while (/* Stop: while x != 0 */) {

The do-while Loop

Use do-while when the body must run at least once before checking the condition. The condition is evaluated at the end.

Shape

// Start
int n = 0;
do {
    std::cout << "n = " << n << "\n";
    n++;                // Step
} while (n < 3);        // Stop (checked after the body)
// Prints n = 0, n = 1, n = 2

Annotated

  • Start: set initial state before the loop
  • Stop: condition at the end (while (...);)
  • Step: usually inside the block, before the condition

Great For

  • Menus that should display at least once
  • Input validation that asks again if invalid

Pitfalls

  • Semicolon required after the condition: } while (cond);
  • Easy to forget Step → potential infinite loop
  • Body always runs once—even if the condition is false initially

Try It!

Show a simple menu at least once; repeat until the user selects “Exit”.

int choice;
do {
    // print menu
    // read choice
    // handle choice
} while (/* Stop: choice != Exit */);

When would a do-while be clearer than a while?

Choosing the Right Loop
  • Use for when Start-Stop-Step are clean and numeric (counting loops).
  • Use while when you repeat “while a condition holds” and Step depends on the body.
  • Use do-while when you must run the body at least once (e.g., menus, input loops).
Challenge

Rewrite the same task three ways (for/while/do-while): print numbers from 10 down to 1.

// for
for (int i = 10; i >= 1; --i) { std::cout << i << " "; }

// while
int i = 10;
while (i >= 1) { std::cout << i << " "; --i; }

// do-while
int j = 10;
do { std::cout << j << " "; --j; } while (j >= 1);
Discussion: Which version is most readable and why?
Common Loop Bugs (and How to Avoid Them)
  • Off-by-one errors: double-check whether you need < or <=.
  • Infinite loops: ensure your Step actually moves toward Stop.
  • Changing the loop variable inside the body: keep updates consistent and intentional.
  • Shadowing variables: avoid re-declaring loop counters in nested scopes inadvertently.
Quick Fix Patterns
  • Write out Start - Stop - Step as comments before coding the loop.
  • Print the loop variable during debugging to observe Step/termination.
  • Use meaningful variable names (e.g., index, count) when clarity matters.
